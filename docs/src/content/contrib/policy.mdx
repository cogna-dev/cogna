# 策略模型

本页聚焦策略（policy）在仓库中的职责和边界，供贡献者进行规则开发与维护时参考。

## 策略职责

策略层负责对变更进行语义判定，例如：

- 哪些变更属于破坏性变更；
- 哪些变更需要阻断发布；
- 哪些变更仅产生提示而不阻断流程。

策略本身不负责抽取源码事实，也不负责生成 diff；它消费上游结果并给出决策。

## 输入与输出约定

贡献策略相关能力时，需要保持以下约定稳定：

- 输入基于 build / diff 产物；
- 输出可映射到 `check` 结果与 SARIF；
- 规则标识、级别和消息应具备可追踪性。

## 贡献策略时的检查点

- 规则命名清晰且可维护；
- 新规则与现有规则没有冲突或重复覆盖；
- 错误消息可被用户理解，并能指向修复方向；
- 文档同步更新，避免“规则上线但文档缺失”。

## 回归建议

```bash
moon check
moon test
```

如果你改动了策略相关文档或规则语义，也应同步检查用户文档中的策略章节是否需要更新。
