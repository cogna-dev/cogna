# 命令总览

CodeIQ 的 CLI 围绕一条稳定工作流展开：先初始化配置，再构建 bundle，随后根据需要执行差异比较、策略检查、查询与发布。当前 `R5-R9` baseline 已完成；当前 active release `R10` 已把 Registry 切换到真实 server capability，并完成 `src/lib/registry` / `src/registry` / `src/cmd/registry` 三层拆分与 async HTTP client/server 落地；当前剩余收口重点主要是 MCP runtime 与文档/验证证据对齐。

| 命令 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| `codeiq init` | 无 | `CIQ Config` | 初始化本地配置与缓存 |
| `codeiq build <repo>` | 代码仓库或策略仓库 | `CIQ Bundle` 或 `CIQ OPA Bundle` | 构建声明快照或策略 bundle |
| `codeiq diff <repo> [--since <ref>] [--test-changes <bool>]` | 代码仓库 | `CIQ Diff` | 生成声明层差异（可附带测试变化区段） |
| `codeiq check <diff-file> --policy <policy-bundle>` | `CIQ Diff` + 本地 `CIQ OPA Bundle` | `SARIF` | 检查此次变更是否符合 policy |
| `codeiq query <query-file>` | 本地 `CIQ Bundle` + `CIQ Query` | `CIQ Result` | 执行一次性本地查询 |
| `codeiq publish <bundle-file>` | 本地 bundle 文件 | publish result | 上传 bundle 到 Registry |
| `codeiq registry upload-local <upload-request-file>` | `codeiq-registry-upload/v1` 请求文件 | `codeiq-publish-receipt/v1` | 将 bundle 写入本地 registry index/bundles |
| `codeiq registry download-local <purl>` | PURL | `codeiq-registry-download/v1` | 解析本地 registry 元数据（含 ETag/304 语义） |
| `codeiq registry download <purl> --output <dir>` | PURL + 输出目录 | 物化后的 bundle 文件集 | 通过本地缓存/远端 Registry 下载并展开 bundle |
| `codeiq registry serve [--port <port>]` | 端口 | 本地 Registry HTTP 服务 | 以前台方式提供 `/health`、publish、resolve、versions、bundle detail、download |
| `codeiq mcp <start\|serve\|stop\|status>` | 无 | 本地 MCP 服务状态 | 管理本地查询代理 |

## `codeiq init`

```bash
codeiq init
```

用于准备本地运行所需的基础状态，例如配置文件、缓存目录，以及后续 CLI 工作流的初始环境。

## `codeiq build <repo>`

```bash
codeiq build ./sdk
codeiq build ./policy
```

`build` 是 CodeIQ 的起点。它把代码仓库或策略仓库构建成标准化产物：

- 代码仓库 -> `CIQ Bundle`
- 策略仓库 -> `CIQ OPA Bundle`

这一步输出的是可发布、可比较、可查询的 bundle，而不是单纯的“索引文件”。

## `codeiq diff <repo>`

```bash
codeiq diff ./sdk
codeiq diff ./sdk --since 2026-03-01T00:00:00Z --test-changes true
```

`diff` 用来比较当前仓库与基线版本，输出 `CIQ Diff`。它关注的是声明层的变化，例如新增、删除、变更、弃用，以及对应的兼容性影响。

当前已支持两类增强参数：

- `--since <ref>`：把比较起点写入 diff 输出中的 `since` 字段。
- `--test-changes true`：在 diff 输出中附加 `includeTestChanges`、`testSummary`、`testChanges` 区段。

## `codeiq check <diff-file> --policy <policy-bundle>`

```bash
codeiq check ./dist/tokio.diff.json --policy ./dist/compat-policy.ciq.tgz
```

`check` 的输入是 `CIQ Diff` 与本地 `CIQ OPA Bundle`，目标是确认这次变更是否符合组织 policy。当前实现会调用本机 `opa eval --bundle --input --format json` 获取 policy decision，并把 decision 合并映射到 `SARIF`（包含 rules/results），便于接入 CI、代码审查与质量平台。

## `codeiq query <query-file>`

```bash
codeiq query ./query.json
```

如果你只需要做一次性查询，那么直接使用 CLI 即可。查询文件中会包含 PURL 和 selector，CodeIQ 会从本地 bundle 中返回对应的 `CIQ Result`。

## `codeiq publish <bundle-file>`

```bash
codeiq publish ./dist/tokio@1.43.0.ciq.tgz
```

`publish` 会把本地构建好的 bundle 上传到本地 registry 存储（`.codeiq/registry`），或在配置 `CODEIQ_REGISTRY_BASE_URL` 时走 remote / loopback HTTP Registry 路径。当检测到同一 PURL 且 `If-None-Match` 命中时，会返回 `304` 语义并产出 `status = "already-exists"` 的 receipt；否则正常上传并返回 `published`。返回结果是 `codeiq-publish-receipt/v1`，包含 `publishId`、`bundleId`、`purl`、`digest`、`bundleType`、`downloadUrl` 与 `status`。

当前 `R8` 的 trust baseline 已完成并冻结：

- verifier 继续保留为 shell OpenSSL CLI（依赖 `CODEIQ_SIGNATURE_PUBLIC_KEY_FILE`）；
- verification failure 继续降级到 `untrusted`；
- `local registry`、`inproc://registry` 与 loopback 开发场景允许免认证 / 降级后继续运行；
- 非本地 remote registry 仍会把 `untrusted` 视为拒绝条件。

当前产品主线工作流是 `init → build → diff → check → query → MCP review`。`publish ↔ registry` 的单机、免认证 baseline 已在 R6 收口完成；`R8` 已把 signature / verifier / downgrade 边界正式写实并固定下来；`R9` 已把 remote auth、transport security 与最小 deployment / ops 边界收口为当前 baseline。

## Remote Registry Auth / Transport Baseline（R9）

当你把 `CODEIQ_REGISTRY_BASE_URL` 指向非本地 remote registry 时，当前 CLI 约定如下：

- 默认要求 remote endpoint 与 remote `downloadUrl` 使用 `https://`；
- 如果确实需要在开发环境中使用非本地 `http://`，必须显式设置：
  - `CODEIQ_REGISTRY_ALLOW_INSECURE_HTTP=true`
- Registry remote transport 现已通过 async HTTP runtime 执行，不再 shell out `curl`。
- 因此下列旧 transport env 现在属于 #strong[不支持且 fail-closed 的配置]：
  - `CODEIQ_REGISTRY_CURL_MAX_TIME`
  - `CODEIQ_REGISTRY_CURL_CONNECT_TIMEOUT`
  - `CODEIQ_REGISTRY_CURL_RETRY`
  - `CODEIQ_REGISTRY_CURL_INSECURE_TLS`
  - `CODEIQ_REGISTRY_TLS_CA_FILE`
  - `CODEIQ_REGISTRY_TLS_CLIENT_CERT_FILE`
  - `CODEIQ_REGISTRY_TLS_CLIENT_KEY_FILE`

### Remote Header Injection

remote registry 的 auth / token / API key 通过环境变量注入。约定是：

- `CODEIQ_REGISTRY_HEADER_<Header-Name>`
- 如果 header 名称里有 `-`，在 env var 中用 `_` 代替，运行时会自动转回 `-`

例如：

```bash
export CODEIQ_REGISTRY_BASE_URL="https://registry.example.test"
export CODEIQ_REGISTRY_HEADER_Authorization="Bearer <token>"
export CODEIQ_REGISTRY_HEADER_X_API_KEY="demo-key"
codeiq publish ./dist/tokio@1.43.0.ciq.tgz
```

### Remote HTTP 开发模式示例

```bash
export CODEIQ_REGISTRY_BASE_URL="http://registry.example.test"
export CODEIQ_REGISTRY_ALLOW_INSECURE_HTTP=true
export CODEIQ_REGISTRY_HEADER_Authorization="Bearer <token>"
codeiq publish ./dist/tokio@1.43.0.ciq.tgz
```

如果没有设置 `CODEIQ_REGISTRY_ALLOW_INSECURE_HTTP=true`，CLI 会拒绝非本地 remote `http://` endpoint 或 `downloadUrl`。

## `codeiq registry upload-local <upload-request-file>`

```bash
codeiq registry upload-local ./dist/upload.json
```

用于把 `codeiq-registry-upload/v1` 请求写入本地 registry。写入成功后返回 `codeiq-publish-receipt/v1`，并把 bundle/索引落在 `.codeiq/registry/bundles` 与 `.codeiq/registry/index`。

## `codeiq registry download-local <purl>`

```bash
codeiq registry download-local "pkg:openapi/acme/payment-api@2026.03.01"
```

按 PURL 从本地 registry 解析下载响应（`codeiq-registry-download/v1`）。该响应包含 `status`、`etag`、`downloadUrl`、`notModified` 与 `signature` 字段，可用于单机 304/缓存语义验证，以及当前 trust baseline 的 metadata 传递。

## `codeiq registry download <purl> --output <dir>`

```bash
codeiq registry download "pkg:openapi/acme/payment-api@2026.03.01" --output ./out
```

在解析到本地 bundle 后，将其物化到输出目录，得到 `manifest.json`、`declarations.ndjson`、`symbols.ndjson`、`metadata.json`、`checksums.txt`。

## `codeiq registry serve [--port <port>]`

```bash
codeiq registry serve
codeiq registry serve --port 8787
```

以当前仓库工作目录为根，启动一个真实的本地 Registry HTTP server。当前最小公开接口为：

- `GET /health`
- `POST /api/v1/bundles`
- `GET /api/v1/bundles/resolve?purl=...`
- `GET /api/v1/packages/{purl-encoded}/versions`
- `GET /api/v1/bundles/{bundleId}`
- `GET /api/v1/bundles/{bundleId}/download`

它会复用 `.codeiq/registry/index` 与 `.codeiq/registry/bundles` 作为本地存储，因此既可以作为 loopback 开发 server，也可以作为 `publish` / `registry download` 的真实 HTTP 证据路径。

## `codeiq mcp <start|serve|stop|status>`

```bash
codeiq mcp start
codeiq mcp serve
codeiq mcp status
codeiq mcp stop
```

MCP 是本地常驻查询代理，只负责高频、内存中的查询能力，也是当前主业务流程里的 review consumption 入口：

- `start`：启动本地 MCP 服务
- `serve`：以前台 HTTP JSON-RPC 方式运行 MCP server，公开 `/health` 与 `/rpc`
- `status`：检查当前服务状态与 health
- `stop`：停止本地 MCP 服务

`start / status / stop` 当前管理的也是同一套 HTTP runtime；`--stdio` 仅保留兼容参数，不再作为 detached stdio 终态。

它不负责 `build`、`diff`、`check`、`publish`。这些操作仍应通过 CLI 完成。

## MCP 查询工具

MCP 当前保留两个工具：

- `codeiq.query.outlines`：根据 PURL 获取包的公开接口大纲，适合先浏览有哪些公开内容可供查询。
- `codeiq.query.symbol`：对某个具体 path 执行精确查询，返回声明、签名、注释、位置与契约摘要。

推荐顺序是：先调用 `codeiq.query.outlines` 获取大纲，再调用 `codeiq.query.symbol` 获取精确信息。

## 推荐组合

### 本地验证

```bash
codeiq init
codeiq build ./sdk
codeiq mcp start
```

### 发布前门禁

```bash
codeiq build ./sdk
codeiq diff ./sdk
codeiq check ./dist/tokio.diff.json --policy ./dist/compat-policy.ciq.tgz
```

### 团队共享

```bash
codeiq build ./sdk
codeiq publish ./dist/tokio@1.43.0.ciq.tgz
```
