# 发版检查

在发布 SDK、API 或 Terraform module 的新版本之前，你需要回答三个问题：

1. **这次改了什么？** — 有哪些公开声明发生了变化
2. **哪些变化可能有风险？** — 是否有 breaking change 会影响调用方
3. **是否违反了团队的兼容性策略？** — policy 检查结果如何

CodeIQ 通过 `build → diff → check` 三步流程帮你系统地回答这三个问题。

---

## 完整流程

```bash
# 1. 构建当前版本的 bundle
cd ./sdk
codeiq build

# 2. 准备 dist/base 与 dist/target 两套产物后做对比
codeiq diff

# 3. 执行兼容性检查
codeiq check
```

### 目前的 diff 前提

当前 `codeiq diff` 不从 CLI 接收 `--since` 或 `--test-changes`。你需要：

- 先把基线版本产物放到 `dist/base/`
- 再把当前版本产物放到 `dist/target/`
- 如果要纳入测试变化，则在 `codeiq.yml -> diff.includeTestChanges` 中开启

## 读懂 diff.json

`diff.json` 是 `codeiq diff` 的核心输出，包含所有检测到的变化。

### 关键字段

| 字段 | 含义 |
|------|------|
| `summary` | 变更摘要（添加/修改/删除的数量） |
| `changes[*].kind` | 变化类型：`added` / `changed` / `removed` |
| `changes[*].level` | 风险级别：`error` / `warning` / `note` |
| `changes[*].path` | 变化的声明路径 |
| `changes[*].semanticDiff.findings` | 具体的语义发现，包含 ruleId 和说明 |
| `changes[*].beforeRecord` | 变化前的源文件位置 |
| `changes[*].afterRecord` | 变化后的源文件位置 |
| `componentChanges` | 软件组件（依赖）的变化 |

### 示例

```json
{
  "summary": {
    "added": 2,
    "changed": 1,
    "removed": 0
  },
  "changes": [
    {
      "kind": "changed",
      "level": "error",
      "path": "Client.Do",
      "semanticDiff": {
        "findings": [
          {
            "ruleId": "compat.go.signature-changed",
            "level": "error",
            "message": "function 'Client.Do' added parameter 'ctx context.Context'"
          }
        ]
      },
      "beforeRecord": { "uri": "client/client.go", "startLine": 87 },
      "afterRecord": { "uri": "client/client.go", "startLine": 89 }
    }
  ]
}
```

## 读懂 check.sarif.json

`check.sarif.json` 是 policy 检查的输出，把 diff 中的 findings 映射为标准 SARIF 格式。

### 优先看哪些字段

| SARIF 字段 | 含义 |
|------------|------|
| `level: "error"` | 高风险，通常会阻止发版 |
| `level: "warning"` | 需要人工确认 |
| `level: "note"` | 提示信息 |
| `ruleId` | 触发的 policy 规则 |
| `message.text` | 对变化的具体说明 |
| `artifactLocation.uri` + `startLine` | 变化在哪个文件哪一行 |
| `tool.driver.rules[].helpUri` | 规则文档链接 |

## 适合哪些发版场景

| 场景 | 建议 |
|------|------|
| SDK / 库的语义化版本发布 | 用 `diff --since` 确认 breaking change，决定是 major / minor / patch |
| OpenAPI 新版本发布 | 检测 parameter requiredness、response status code 等高风险变化 |
| Terraform module 新版本 | 检测 variable 必填性和 output 删除 |
| PR 合并前的门禁检查 | 接入 CI，自动运行 diff + check |
