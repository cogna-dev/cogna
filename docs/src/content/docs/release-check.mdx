# 发版前兼容性检查

如果你是 SDK、库、Terraform Module 或 OpenAPI 的维护者，这通常会是你最关心的一页。

CodeIQ 在这个场景里解决的核心问题不是“找所有改动”，而是：#strong[在发布前，把公开接口和契约变化整理成一份可检查、可解释、可进入审查流程的结果]。

## 最短路径

```bash
codeiq build ./sdk
codeiq diff ./sdk
codeiq check ./dist/tokio.diff.json --policy ./dist/compat-policy.ciq.tgz
```

这三步分别回答三个不同的问题：

- `build`：当前版本到底公开了什么？
- `diff`：和基线相比，到底变了什么？
- `check`：这些变化是否违反了本地 policy？

## 为什么不能只看源码 diff？

源码 diff 更适合看“实现上改了哪些行”，但发布前兼容性检查更关心的是：

- 某个公开函数是不是被删了；
- 某个 method 是否变成了更危险的版本；
- 某个 OpenAPI parameter 是否从 optional 变成了 required；
- 某个软件成分是不是发生了版本升级；
- 这些变化里哪些只是 note，哪些已经值得 warning 或 error。

CodeIQ 的 `CIQ Diff` 就是在回答这个问题。

## `CIQ Diff` 里最值得先看的字段

对于用户来说，不需要先看完所有 JSON 细节。优先关注这些内容：

### `changes[*].kind`

告诉你这是新增、删除、修改还是弃用。

### `changes[*].level`

告诉你这是更偏向 breaking、warning 还是 note。

### `changes[*].semanticCategory`

这是当前更有价值的字段：它不是只告诉你“改了”，而是告诉你“改成了什么类型的变化”。

### `changes[*].beforeRecord` / `afterRecord`

如果你要追查具体证据，这里能帮助你看到变化前后的声明记录，而不是只看一条抽象描述。

### `componentChanges`

如果你关心的是软件成分变化、版本升级或依赖结构变化，这部分是重点。

## `check` 做了什么？

`check` 不会重新分析仓库。它做的是：

1. 读取 `CIQ Diff`；
2. 把 diff 整形成 OPA-friendly 输入；
3. 调用本地 `opa eval --bundle ...`；
4. 把决策结果合并为一份本地 `SARIF` 输出。

因此，`check` 的价值在于把“结构化变更事实”转换成“可审查、可门禁”的结果层。

## 如何读 `SARIF`？

第一次使用时，你可以把它简单理解成三层信号：

- `error`：高风险，通常表示需要阻断或人工明确审批；
- `warning`：值得重点检查，但未必必须阻断；
- `note`：信息性结果，帮助你理解变化背景。

当前 CodeIQ 的文档口径应理解为：#strong[本地 minimal SARIF v2.1.0 output]。它足够支持当前本机工作流，但不应被误解为“已经承诺所有 hosted 平台都能零调整直接消费”。

## 什么时候这条路径最值得接入？

你很适合优先接入这条路径，如果你满足以下任一情况：

- 你经常发版 SDK / Library；
- 你维护的 OpenAPI 变化需要被团队审查；
- 你希望把 Terraform / OpenAPI / 代码接口治理统一到一套工作流；
- 你不想只靠人肉 review 来判断这次改动是否破坏兼容性。
