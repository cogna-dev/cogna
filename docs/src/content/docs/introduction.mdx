# CodeIQ 能帮我解决什么问题？

CodeIQ 解决的不是“怎么读懂所有实现细节”，而是另一个更常见的问题：当你要发版、升级依赖、审查 OpenAPI 或让 AI 查询第三方库时，怎样才能先拿到一份 #strong[稳定、可追溯、可比较] 的事实基础。

它把仓库里的公开接口、配置契约和 API 合约整理成标准化产物，让你可以在本机完成这几类高价值动作：

- 发布前先看清这次到底改了什么；
- 把变更结果交给 policy 检查，而不是只看文本 diff；
- 让 AI / Agent 按稳定入口查询依赖和 API，而不是盲搜全仓；
- 把同一份 bundle 复用给 CLI、MCP 和自动化脚本。

## 它最适合哪些场景？

### 1. SDK / Library 发版前检查

你要确认的不只是“代码过没过测试”，还包括：

- 是否删掉了公开 API；
- 是否把某个接口变成了更危险或不兼容的版本；
- 这些变化是否违反组织自己的兼容性 policy。

### 2. Terraform / OpenAPI 变更治理

如果你的仓库不是传统代码，而是 Terraform Module 或 OpenAPI 规格，CodeIQ 仍然能把这些契约变化整理成结构化产物，而不是让你手工对 YAML / HCL 做逐行比较。

### 3. 让 AI 查询第三方依赖与 API

如果你已经在用 AI 写代码，你会很快遇到一个问题：AI 很容易找到“看起来像答案的文本”，但不一定能稳定找到“当前这个版本真正公开了什么”。

CodeIQ 的做法是：

- 先给 AI 一个 #strong[outlines] 级别的大纲入口；
- 再让它对具体 symbol 做精确查询；
- 最后把路径、签名、位置、契约摘要都绑定到 bundle 里，而不是漂浮在文档片段里。

## 它和 README、grep、LSP、lint 有什么不同？

| 方式 | 更擅长什么 | 局限是什么 |
| --- | --- | --- |
| README / 外部文档 | 快速理解产品背景 | 文档可能过时，和当前实现不一致 |
| grep / 全仓搜索 | 找文本、找关键词 | 很难稳定回答“这次发布到底改了哪些公开契约” |
| LSP / IDE 导航 | 交互式跳转和定位 | 不会自动给你可发布、可比较、可审查的产物 |
| lint / 静态分析 | 实现级质量问题 | 不以公开接口变化作为主要输出 |
| #strong[CodeIQ] | 公开接口、结构化契约、变更检查、AI 查询复用 | 当前只覆盖单机 local-only 工作流 |

## CodeIQ 给你的直接产物是什么？

| 产物 / 能力 | 你会直接得到什么 |
| --- | --- |
| `CIQ Bundle` | 一份可发布、可查询、可比较的仓库快照 |
| `CIQ Diff` | 一份面向声明和软件成分的结构化变更结果 |
| `SARIF` | 一份可被审查流程或 CI 消费的检查输出 |
| Local Registry | 一套共享本地 bundle store 和 façade |
| local query runtime | 一个面向 AI / Agent 的交互式查询入口 |

## 典型工作流是什么？

你不需要一开始就理解所有概念。只要先记住这条主线：

```text
init -> build -> diff -> check -> query / mcp -> publish（可选）
```

- `init`：准备配置；
- `build`：把仓库整理成 bundle；
- `diff`：看当前版本与基线版本之间的结构化变化；
- `check`：把 diff 交给本地 policy bundle 做检查，并输出 SARIF；
- `query / mcp`：消费 bundle，给脚本或 AI 查询；
- `publish`：把 bundle 放进本机共享 store，供后续复用。

## 当前边界是什么？

这点非常重要：CodeIQ 当前是 #strong[single-machine / local-only] 产品边界。

这意味着：

- 面向本机上的 CLI、Local Registry 和 local query runtime；
- 不承诺 hosted registry、distributed sync、多节点 HA；
- 不承诺 trust verifier、KMS、远端签名验证链路；
- `MCP` 在这里是本地查询入口，不是构建、发版或门禁执行入口。
