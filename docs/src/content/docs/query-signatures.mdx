# 检索函数签名

你在做代码升级评估时，经常需要知道某个函数的参数列表是否变了，某个 Terraform variable 是否还是可选的，或者某个 OpenAPI operation 的 response schema 是什么。与其翻源码或文档，你可以直接用 Cogna 查询本地 bundle 里的准确信息。

---

## 两种查询方式

| 场景 | 方式 |
|------|------|
| 查一个具体的符号 | `cogna query <query-file>` |
| 让 AI 自主探索和回答 | `cogna mcp serve --port 3000` |

---

## 方式一：CLI 一次性查询

### 步骤 1：构建 bundle

```bash
cd ./sdk
cogna build
```

### 步骤 2：准备查询文件

```json
{
  "schemaVersion": "ciq-query/v1",
  "purl": "pkg:cargo/tokio@1.43.0",
  "selector": {
    "path": "tokio::sync::Mutex::lock",
    "kind": "function"
  },
  "intent": "upgrade"
}
```

`intent` 字段说明你查询的目的（`upgrade` / `review` / `understand`），帮助 Cogna 优化返回内容。

### 步骤 3：执行查询

```bash
cd ./sdk
cogna query ../query.json

# 结果保存在
cat ./query.result.json
```

---

## 方式二：MCP 让 AI 探索

当你需要让 AI 助手回答"这个 SDK 里有没有线程安全的 Mutex？" 这类问题时，MCP 更适合。

```bash
cd ./sdk
cogna build
cogna mcp serve --port 3000
```

建议 AI 按以下顺序调用：

1. **`cogna.query.outlines`** — 先获取所有公开路径的大纲
2. **`cogna.query.symbol`** — 再精确查询某个路径的签名和文档

这个顺序的好处是：先缩小范围，避免 AI 猜错路径。

### 示例：查询 Rust 函数签名

AI 调用 `cogna.query.symbol`：

```json
{
  "purl": "pkg:cargo/tokio@1.43.0",
  "path": "tokio::sync::Mutex::lock",
  "kind": "function"
}
```

返回结果：

```json
{
  "path": "tokio::sync::Mutex::lock",
  "kind": "function",
  "signature": "pub async fn lock(&self) -> MutexGuard<'_, T>",
  "location": { "uri": "src/sync/mutex.rs", "startLine": 234 },
  "docs": "Locks this mutex, causing the current task to yield until the lock has been acquired.",
  "language_specific": {
    "is_async": true,
    "is_unsafe": false,
    "receiver": "&self"
  }
}
```

## 各生态查询结果的特殊字段

不同语言/规格的 `language_specific` 字段有所不同：

### Go

```json
{
  "language_specific": {
    "receiver": "*Client",
    "is_pointer_receiver": true,
    "method_set": ["interface.Doer"]
  }
}
```

### Rust

```json
{
  "language_specific": {
    "is_async": true,
    "is_unsafe": false,
    "where_clause": "T: Send",
    "receiver": "&self"
  }
}
```

### Terraform

```json
{
  "language_specific": {
    "kind": "variable",
    "type": "string",
    "is_required": true,
    "default": null
  }
}
```

### OpenAPI

```json
{
  "language_specific": {
    "method": "GET",
    "path": "/users/{id}",
    "parameters": [
      { "name": "id", "in": "path", "required": true }
    ]
  }
}
```

## 实际使用场景

### 场景一：升级前确认函数签名是否变化

```bash
# 先构建新版本 bundle
cd ./sdk
cogna build

# 准备 dist/base 与 dist/target 两套产物后，用 diff 找出有哪些签名变了
cogna diff

# 再用 query 精确查询某个函数的当前签名
cogna query ../query.json
```

当前 CLI 不支持 `diff --since ...` 这种覆盖式参数；版本基线由你准备好的 `dist/base` / `dist/target` 决定。

### 场景二：让 AI 帮你判断某次升级是否安全

```bash
cd ./sdk
cogna build
cogna mcp serve --port 3000
# 让 AI 调用 outlines 列出变化的函数，再用 symbol 一一确认
```
