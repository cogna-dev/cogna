# Provider 覆盖率报告

这页只回答一个问题：CodeIQ 当前到底能从哪些语言 / 生态里提取哪些声明事实，以及这些能力现在属于 `LSP Extractor` 还是 `CodeIQ Extractor`。

这里的结论只基于当前 repo 证据：`src/cmd/build/extractor.mbt`、`src/schema/declarations.mbt`、`src/e2e/lsp_extractor_test.mbt`、`examples/*/snapshots/lsp/*`、以及 `spec/A001-lsp-extractor.typ` 中定义的完整语法覆盖面。

## 总览矩阵

| Profile | 当前 build 模式 | `LSP Extractor` | `CodeIQ Extractor` | 当前结论 |
| --- | --- | --- | --- | --- |
| `go-module` | `lsp-assisted` + `gopls` | ✅ 当前主 baseline | ⚪ 规划中 | Go 目前按 LSP baseline 对外表述 |
| `rust-crate` | `lsp-assisted` + `rust-analyzer` | ✅ 当前主 baseline | ⚪ 规划中 | Rust 目前按 LSP baseline 对外表述 |
| `terraform-module` | `parser-first` + `hcl-parser` | — | ✅ 当前实现 | Terraform 只按 CodeIQ Extractor 记账 |
| `openapi-spec` | `parser-only` + `none` | — | ✅ 当前实现 | OpenAPI 只按 CodeIQ Extractor 记账 |

> 说明：`src/cmd/build/extractor.mbt` 当前的 `profile_mode(...)` 明确把 Go / Rust 标成 `lsp-assisted`，把 Terraform / OpenAPI 标成 `parser-first` / `parser-only`。因此本页把 Terraform 和 OpenAPI 只放进 `CodeIQ Extractor` 列，不把它们描述成 LSP 生态。

> 表格中的状态口径：`✅ 已验证` 表示当前 repo 已有 snapshot / test 证据；`🟡 部分` 表示 spec 已纳入或当前只覆盖一部分表现；`⚪ 规划中 / 未覆盖` 表示当前 repo 还没有直接证据。

## 当前可稳定写入 bundle 的声明事实

无论 provider 来自哪条抽取路径，只要进入当前 bundle，就会落到同一份声明 contract。`src/schema/declarations.mbt` 当前要求的核心字段包括：

- `id`
- `kind`
- `name`
- `canonical_name`
- `path`
- `signature`
- `shape`
- `visibility`
- `docs`
- `baselineRole`
- `language`
- `source_refs`
- `relations`
- `location`
- `language_specific`

这意味着本页关注的“覆盖率”不是 IDE 交互功能，而是这些字段能否稳定落盘进 `declarations.ndjson` / `symbols.ndjson` / `metadata.json`。

## 与官方生态的关系

本页主体以 repo 现状为准，但 provider 选择也和各自官方生态一致：

- Go：`gopls` 官方能力面包含 definition、references、implementation、typeDefinition、hover、documentSymbol、signatureHelp 等，适合作为 LSP baseline。
- Rust：`rust-analyzer` 官方能力面同样覆盖 goto definition / implementation / type definition、hover、document symbol、signature help、references。
- Terraform：HashiCorp 的语言与工具链更自然地围绕 HCL parser / config inspection 建模，而不是把 LSP 当成声明抽取真值源。
- OpenAPI：OpenAPI Specification 本身就是结构化 contract，天然适合 spec-driven extraction。
