# 第一次跑通：10 分钟完成 build → diff → check → query

如果你第一次接触 CodeIQ，最好的起点不是先把所有概念读完，而是先在一个熟悉的仓库上跑通一条最短路径。

这条路径的目标不是“学会所有功能”，而是确认三件事：

1. 你的仓库能否稳定产出 bundle；
2. 你能否看懂结构化 diff 和检查结果；
3. 你能否从 bundle 拿到一次真实查询结果。

## 前置条件

开始之前，建议你准备：

- 一个熟悉的代码仓库或规格仓库；
- 本地可用的 `codeiq` CLI；
- 如果要执行 `check`，本机还需要可用的 `opa` CLI；
- 一个本地 policy bundle（如果你暂时没有，也可以先只跑到 `diff` 和 `query`）。

## 第 1 步：初始化

```bash
codeiq init
```

初始化的目标是生成后续工作流所需的基础配置，例如 `codeiq.yaml`、本地 cache 目录和默认的检查配置。

你现在不需要理解所有字段，只需要知道：这一步在告诉 CodeIQ “这个仓库是什么、该如何构建、后续检查默认应该怎么接”。

## 第 2 步：构建 bundle

```bash
codeiq build ./sdk
```

如果这是代码仓库，你会得到一个 `CIQ Bundle`；如果这是策略仓库，你会得到一个 `CIQ OPA Bundle`。

在当前实现里，代码 bundle 通常会带上这些关键产物：

- `declarations.ndjson`
- `symbols.ndjson`
- `software-components.ndjson`
- `sbom.cdx.json`
- `sbom.spdx.json`

第一次体验里，你只需要确认 `dist/` 下已经出现了这些结果中的主要部分。

## 第 3 步：生成 diff

```bash
codeiq diff ./sdk
codeiq diff ./sdk --since 2026-03-01T00:00:00Z --test-changes true
```

`diff` 生成的不是源码 patch，而是一份面向声明和软件成分的结构化变更结果。

第一次体验时，优先看下面几类字段：

- `added / removed / changed / deprecated`
- `semanticCategory`
- `semanticDiff`
- 如果启用了 `--test-changes true`，再看 `testSummary` 和 `testChanges`

## 第 4 步：执行 check（可选但强烈推荐）

```bash
codeiq check ./dist/tokio.diff.json --policy ./dist/compat-policy.ciq.tgz
```

如果你本机有可用的 policy bundle，这一步最能体现 CodeIQ 的价值：

- 它会把 `CIQ Diff` 整形成 OPA-friendly 输入；
- 再通过本地 `opa eval --bundle ...` 执行策略判断；
- 最终输出一份本地 `SARIF`，给审查和 CI 消费。

你现在不需要先读懂全部 `SARIF` 字段。第一次体验只需要知道：CodeIQ 已经把“这次变更需不需要关注”整理成 `error / warning / note` 这样的结果层级。

## 第 5 步：做一次查询

### 方式 A：一次性 CLI 查询

准备一个简单的 `query.json`：

```json
{
  "schemaVersion": "ciq-query/v1",
  "purl": "pkg:cargo/tokio@1.43.0",
  "selector": {
    "path": "tokio::sync::Mutex::lock",
    "kind": "function"
  },
  "intent": "upgrade"
}
```

然后执行：

```bash
codeiq query ./query.json
```

### 方式 B：交互式查询

```bash
codeiq mcp start
```

如果你是给 AI 或 Agent 使用，推荐顺序是：

1. 先调用 `codeiq.query.outlines`；
2. 再对感兴趣的 path 调用 `codeiq.query.symbol`。

## 什么叫“跑通了”？

你完成第一次成功路径后，至少应该看到下面这些结果中的大部分：

- 已生成 `codeiq.yaml`；
- `dist/` 下已经出现 bundle 相关产物；
- 已生成一份 `CIQ Diff`；
- 如果执行了 `check`，已生成一份本地 `SARIF`；
- 你已经能通过 `query` 或 `mcp` 拿到一次真实查询结果。

## 第 6 步：如果需要，再做本地共享

```bash
codeiq publish ./dist/tokio@1.43.0.ciq.tgz
```

这不是第一次体验的必选项，但如果你希望同机上的多个命令、脚本或 AI 工作流复用同一份 bundle，就应该把它写入本地共享 store。
