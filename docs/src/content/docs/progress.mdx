## R5 · Registry Boundary & Trust Hardening（Active）

**Scope**

- 只覆盖上一版明确顺延的两类问题：
  - registry remote 主证明仍主要依赖 env stub；
  - signature path 仍停留在 shell OpenSSL CLI 路径，non-shell verifier 需要正式阶段结论。

**Done when**

- registry remote path 具备 localhost HTTP fixture e2e，且不再把 `CODEIQ_REGISTRY_REMOTE_*_RESPONSE_JSON` 作为主证明；
- shell OpenSSL CLI 的当前地位、non-shell verifier 的后续决策、以及 `publish / registry` 的 verify 口径在 docs / plan / tests 中保持一致；
- `spec/plan.typ` 与本页对 R5 状态、verify、carry-forward 保持同步。

**Repo evidence（当前已可复用的基础）**

- `src/cmd/registry/main_test.mbt`
- `src/cmd/registry/download_bundle_test.mbt`
- `src/cmd/publish/main_test.mbt`
- `src/cmd/registry/main.mbt`
- `src/cmd/publish/main.mbt`
- `spec/verify.typ`

**Verify**

- `moon test src/e2e/registry_remote_test.mbt`
- `moon test src/cmd/registry/main_test.mbt`
- `moon test src/cmd/publish/main_test.mbt`
- `pnpm build`（在 `docs/` 下）

**Non-goals**

- 认证鉴权、生产 registry 部署、多租户、KMS / key rotation 平台、完整企业化运维。

**Carries forward**

- 若 R5 后仍要继续推进 shared non-shell verifier implementation、trust root、enterprise ops，统一进入下一版 release。

## R5 收口任务（唯一执行清单）

### R5-1 · Registry localhost HTTP fixture e2e

- **Goal**：补上 repo 内真实 HTTP boundary 的主证明，让 remote registry path 不再主要依赖 env stub。
- **Main touchpoints**：`src/cmd/registry/main.mbt`、`src/cmd/registry/main_test.mbt`、`src/cmd/registry/download_bundle_test.mbt`、`src/e2e/registry_remote_test.mbt`
- **Depends on**：无
- **Done when**：
  - upload / resolve / `304 Not Modified` / cache replay 通过 localhost HTTP fixture e2e；
  - env stub tests 继续保留，但不再承担主证明；
  - docs 与 plan 的 verify 清单统一引用 `src/e2e/registry_remote_test.mbt`。
- **Verify**：
  - `moon test src/e2e/registry_remote_test.mbt`
  - `moon test src/cmd/registry/main_test.mbt`
- **Out of scope**：替换 `curl` transport、引入 auth、部署真实外部 registry。

### R5-2 · Signature path decision record

- **Goal**：把“当前继续使用 shell OpenSSL CLI，non-shell verifier 以后再考虑”升级为正式的阶段性设计记录。
- **Main touchpoints**：`spec/verify.typ`、`spec/plan.typ`、`docs/src/content/docs/progress.mdx`、`src/cmd/registry/main.mbt`、`src/cmd/publish/main.mbt`
- **Depends on**：无
- **Done when**：
  - 文档中明确区分当前 shipped path 与后续 shared verifier 议题；
  - `publish / registry` 的 verify 口径在文档和测试语义里一致；
  - `verify.typ` 对 trusted publishing / hash / signature / 当前选择的解释与 release plan 保持一致。
- **Verify**：
  - `moon test src/cmd/registry/main_test.mbt`
  - `moon test src/cmd/publish/main_test.mbt`
  - `typst compile spec/verify.typ /tmp/codeiq-verify.pdf`
- **Out of scope**：本轮必须落地 non-shell verifier 实现。

### R5-3 · Shared verifier feasibility checkpoint

- **Goal**：在真正开做 non-shell verifier 前，先把实现前提收敛成可执行或可放弃的 checkpoint。
- **Main touchpoints**：`spec/verify.typ`、相关依赖调研记录、必要时 `src/lib/signature/*` 设计草案
- **Depends on**：`R5-2`
- **Done when**：
  - 明确列出可行实现路线、依赖来源、测试前提与风险；
  - 若结论是否定的，也要把“为什么本轮不做”写成正式记录；
  - 结果可以直接支持下一版是否新开 shared verifier implementation release。
- **Verify**：
  - `typst compile spec/verify.typ /tmp/codeiq-verify.pdf`
  - `pnpm build`（在 `docs/` 下）
- **Out of scope**：在没有明确依赖与验证策略前直接硬写 `src/lib/signature/*`。

### R5-4 · Release closeout

- **Goal**：在 `R5-1` 到 `R5-3` 完成后关闭本版计划，而不是继续在同一 release 中追加横向能力。
- **Main touchpoints**：`spec/plan.typ`、`docs/src/content/docs/progress.mdx`
- **Depends on**：`R5-1`、`R5-2`、`R5-3`
- **Done when**：
  - `spec/plan.typ` 与本页都把 R5 标成 completed；
  - verify 清单只引用本轮真实完成的 HTTP / signature-decision 证据；
  - 剩余事项全部移入 backlog / 下一版 plan。
- **Verify**：
  - `pnpm build`（在 `docs/` 下）
  - 相关 MoonBit tests 通过
- **Out of scope**：继续延长当前 release 生命周期。
