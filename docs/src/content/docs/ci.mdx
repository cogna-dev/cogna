# 持续集成

把 Cogna 接入 CI 是最常见的使用场景之一：每次提交 Pull Request 时，自动检测这次变更是否破坏了 API 兼容性，并把结果展示在 PR 页面上。

当前仓库推荐的接入方式是三层组合动作：

- `setup-moonbit`：安装 MoonBit toolchain；
- `setup-cogna`：从当前仓库源码安装 `cogna` CLI，并写入 PATH；
- `run-cogna`：执行 `cogna <command>`，在 `check` 时可选上传 SARIF。

## GitHub Actions 完整示例

下面是一个可直接使用的 GitHub Actions workflow。它假设你的仓库里已经准备好了 `cogna.yml`，并且在运行 `diff` 前会生成 `dist/base` 与 `dist/target`：

```yaml
name: Cogna Compatibility Check

on:
  pull_request:
    branches: [main, release/*]
  push:
    branches: [main]

jobs:
  compat-check:
    name: API Compatibility Check
    runs-on: ubuntu-latest
    permissions:
      contents: read
      security-events: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0   # 需要完整历史才能做 diff

      - name: Setup MoonBit
        uses: ./integrations/actions/setup-moonbit

      - name: Setup Cogna
        uses: ./integrations/actions/setup-cogna

      - name: Build bundle
        uses: ./integrations/actions/run-cogna
        with:
          working-directory: ./sdk
          command: build

      - name: Generate diff
        uses: ./integrations/actions/run-cogna
        with:
          working-directory: ./sdk
          command: diff

      - name: Run compatibility check
        uses: ./integrations/actions/run-cogna
        with:
          working-directory: ./sdk
          command: check
          upload-sarif: true

      - name: Save artifacts
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: cogna-results
          path: |
            ./sdk/dist/diff.json
            ./sdk/dist/check.sarif.json
            ./sdk/dist/bundle.ciq.tgz
```

如果你只想安装 CLI 做 smoke test，也可以先调用：

```yaml
- uses: ./integrations/actions/setup-moonbit
- uses: ./integrations/actions/setup-cogna
- run: cogna init --output /tmp/cogna-smoke.yml
```

## 两种常见 CI 用法

### 用法一：发版前 gate（阻止合并）

在发版 PR 或 release branch 上运行，有 `error` 级别的 breaking change 时让 CI 失败：

```yaml
- uses: ./integrations/actions/run-cogna
  with:
    working-directory: ./sdk
    command: check
    upload-sarif: true
```

`run-cogna` 会保留 `check` 的退出码，因此当有 `error` 级别结果时，workflow 会失败；同时仍然会尝试把 `dist/check.sarif.json` 编码后上传到 GitHub Code Scanning REST API。

### 用法二：仅记录，人工审查（不阻止合并）

如果你的团队还在逐步引入 Cogna，可以先先跑 `build` / `diff` / `check`，但把最终失败策略留给 workflow 外层控制。例如保留 artifact 和 SARIF，而不把这一步设成 merge gate。

```yaml
- name: Run compatibility check for review only
  continue-on-error: true
  uses: ./integrations/actions/run-cogna
  with:
    working-directory: ./sdk
    command: check
    upload-sarif: true
```

这样 PR 页面仍然会出现 SARIF 结果，但这一步不会阻止后续 artifact 保存或人工审查流程。

## 关键约束

- `setup-cogna` 复用当前仓库的本地源码安装语义，而不是依赖远程 install script；
- `run-cogna` 不会替你准备 `dist/base` 与 `dist/target`，这些输入仍需在 `diff` 前准备好；
- 当 `command: check` 且 `upload-sarif: true` 时，workflow 必须具备 `security-events: write` 权限；
- `run-cogna` 默认读取 `working-directory/dist/check.sarif.json`，也可以通过 `sarif-file` 覆盖。

## 用 act 做本地 remote extractor e2e 验证

仓库内已经提供了 `.github/workflows/remote-e2e.yml` 与 `just act-e2e`：

```bash
just act-e2e
```

这会以 `linux/amd64` 容器运行 `act`，执行 remote extractor e2e 的 prepare / run 两阶段：

- setup MoonBit / Cogna / Go / Rust / Terraform
- restore cache（act 本地可复用缓存）
- `scripts/ci/prepare-remote-e2e.sh` 预热依赖
- `scripts/ci/run-remote-e2e.sh` 执行 remote case

注意：`act` 主要用于复现 workflow 编排；它不应被视为 GitHub hosted cache backend 的等价环境。

## CI 产物说明

每次 CI 运行后，建议保存以下产物：

| 文件 | 用途 |
|------|------|
| `./sdk/dist/diff.json` | 结构化变更列表，可供脚本解析或人工阅读 |
| `./sdk/dist/check.sarif.json` | 标准 SARIF，可上传到 GitHub Code Scanning 等平台 |
| `./sdk/dist/bundle.ciq.tgz` | bundle 归档，可通过本地 cache / HTTP cache proxy 复用或分发 |
| `./sdk/.cogna/policies/` | policy 目录；`check` 会直接把它传给 OPA |
